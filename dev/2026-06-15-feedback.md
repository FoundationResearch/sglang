## 一、原始反馈（存档）

> 最近用 sglang 评估下游任务的时候遇到几个问题，可以辛苦排查修复一下么
>
> 1. 对齐一致性的时候需要开启 `headwise_topk_softmax`，此时如果测试序列短于 `topk*chunksize` 的话一致性测试会 fail，但是长序列可以 pass，而且当前 `headwise_topk_softmax=True` 走的是 torch naive 实现，最好也改成走 fused kernel 而不是 torch fallback，避免后面评测下游任务的时候遇到长序列 OOM。这点主要是正确性相关的，评测下游任务时需要走这个路径，不用像 topk max pool 那个路径一样特别抠速度，只要能保证正确性以及避免 torch 部分导致的 OOM 即可。一致性测试的时候可以开启 `headwise_topk_softmax=True` 然后测试一下短序列比如 500，1000 之类的来复现一下，因为大部分下游任务都属于短序列。
>
> 2. `hsa_extend_paged_fwd_kernel` 和 `hsa_swa_extend_kernel` 的长度参数最好不要声明为编译时常量 `tl.constexpr`，否则评测下游任务的时候遇到不同长度的输入会触发重新编译导致变慢，不过修改这个后不确定是否会对速度有影响，最好重新测速一下，我记得 tilelang 里如果 kernel 输入形状定义为编译时常量的话速度会比 dynamic shape 更快一些。
>
> 3. 开启 chunk prefill 后一致性测试会 fail，这一点在之前群里发过的 `hsa_compressedlmkk_37eabce0b_semantics.md` 里第 9 点提到过，但是现在还没修复。一致性测试的时候最好也开 chunk prefill 跑一下。
>
> 4. 还有一个正确性相关的问题，就是目前 reuse cache 开启后 kv cache 可以复用前缀缓存，但是 lmk cache 貌似还不支持跨请求的缓存写入和恢复，具体体现在用两个共享前缀的样本（长度都大于 3000 来避免第 1 点 bug 的影响）单独测试 sglang 推理的时候，开启和不开启 cache reuse 时第 2 个样本的推理结果完全不同，都是开启 cache reuse 后输出就开始一直复读了。这个也比较影响评估下游任务，大部分 fewshot 的任务都需要 cache reuse 来加速。
>
> 【群内补充 a】我初步排查到的导致短序列不一致的一个 bug 是 `sglang/python/sglang/srt/layers/attention/hsa/kernels/internal_swa_decode.py` 里面 `fused_internal_swa_decode_kernel` 的 `alpha = tl.exp(m_i - m_new)` 要改成有保护的 `alpha = tl.where(m_new == float("-inf"), 1.0, tl.exp(m_i - m_new))`。（朋友："我马上修"）
>
> 【群内补充 b】开启 cuda graph 后一致性测试 fail（即使长序列）。朋友定位了根因并给了一版修复补丁（见下方 md 文档）。

### 群内补充 b 的完整文档（HSA CUDA Graph 开启后一致性 fail 修复梳理）

根因：SGLang 捕获 decode graph 时先 dry-run 几次再 capture。HSA decode selection 在 `HSAMetadata` 上缓存的 per-step scratch
（`hsa_per_step_cand_page_ids / cand_mask / slots / all_lmk_k / all_prior_b / hsa_window`）在 dry-run 阶段被 dummy capture input 填充，
真正 capture 时 Python 看到 cache 已存在直接复用 → capture 进 graph 的是"读 stale scratch"而非"按当前 metadata 重算" → replay 用冻结的
candidate/slot/lmk_k → logits 从最早几步就偏。

修复（correctness fix，非 workaround）：在 `_run_selection_decode` 中识别是否为 CG metadata：
```python
cg_buf = getattr(self, "_cg_page_table_1", None)
is_cg_metadata = cg_buf is not None and page_table_1.data_ptr() == cg_buf.data_ptr()
```
CG metadata 时不再读/写这些 Python-side scratch cache，每次 capture 都把 candidate/mask/slots/all-layer LMK gather 的计算捕获进 graph，
replay 时按刷新后的 persistent buffer 重算。eager 路径保留原 cache（性能不受影响）。验证：500/500 100% 一致。
长期更优方案 B：把这些 scratch 改成 CG persistent device buffer + Triton 原地更新（工程量更大）。

## 三、复现核实记录

> 逐条填写：是否复现成功、根因、是真 bug 还是配置/跑法问题、修复方案。

### P6（swa alpha -inf 保护）— 状态：✅ 真 bug（静态坐实，两处）
`internal_swa_decode.py`：
- L130 `alpha = tl.exp(m_i - m_new)`：当某 query 的 SWA 窗口内**无任何有效 KV**（`w_valid` 全 False → `scores` 全 `-inf` → `block_max=-inf`，且初始 `m_i=-inf`）时 `m_new=-inf`，`alpha=exp(-inf-(-inf))=exp(nan)=nan` → `acc=acc*nan=nan` 污染。
- L192 `alpha = tl.exp(m_s - m_global)`（split-W 合并）：当 `m_global=-inf`（所有 split 都空）时同样 nan。**朋友只提了 L130，L192 也要修。**
修复：`alpha = tl.where(m_new == float("-inf"), 1.0, tl.exp(m_i - m_new))`（L130）；L192 用 `tl.where(m_global == float("-inf"), 0.0, ...)`。

### P7（cuda graph per-step cache 污染）— 状态：✅ 真 bug（静态坐实）
`hsa_backend.py:471–532` `_run_selection_decode` 读写 `hsa_per_step_{cand_page_ids,cand_mask,hsa_window,slots,all_lmk_k,all_prior_b}`，**全程无 CG 保护**（全文 grep 无 `is_cg_metadata`）。已有 `_cg_page_table_1` persistent buffer（L880-908）但 selection scratch 不在其列。机制与朋友的根因分析一致：CG dry-run 污染 cache → replay 用 stale。朋友给了补丁（CG metadata 时禁用这些 Python cache）。

### P3（kernel 长度 constexpr 重编译）— 状态：✅ 真 bug（静态坐实）
`hsa_extend.py` 的 `T: tl.constexpr`；`hsa_swa_extend.py` 的 `T/TOTAL_KV/PREFIX_LEN: tl.constexpr`。变长输入 → 每个新长度触发 JIT 重编译。改成运行时参数可消除，但需重测速（constexpr 边界利于 triton loop 展开，可能影响速度）。

### P1（短序列 headwise 一致性）— 状态：✅ 复现 + 根因=P6 + 修复验证 IDENTICAL
真权重（G=1 345M，wikitext 2000 步 loss 6.71）+ `REPRO_HEADWISE=1`：
- **修复 P6 前**：SHORT 10 → PREFILL KL=0.246 [WARN]，**decode KL=nan [FAIL]**，sglang argmax 恒=0（复读，正是朋友现象），0/7。
- **修复 P6 后**：SHORT 63/65/128/256 全部 **[IDENTICAL]**（KL≈2e-5，argmax 100%）。nan/复读消失。
- 唯一残留：**SHORT 10**（10 token < chunk_size 64，连一个完整 chunk 都没有）仍 [WARN]（KL 0.13–0.25）。这是 <1 chunk 的极端边界，真实下游评测（500/1000 token）碰不到；编译期还伴随 tilelang 越界 warning（`merged_indices k[0]>=512` 等），疑似 selection 在不足 1 chunk 时的边界，留待后续。
结论：**P1 即 P6 的短序列表现，P6 修复后实用短序列（≥1 chunk）完全对齐。** tiny 复现不出是因欠训练（输出近均匀，KL 淹没差异）——必须用学过的真权重。

（对照基线：tiny 权重 headwise=True 全 PASS；真权重 maxpool 路径全 PASS；只有"真权重 + headwise=True + 短序列"暴露，与朋友报告完全吻合。）

### P2（headwise torch fallback OOM）— 状态：✅ 确认（设计如此）
`hsa_backend.py:1809` extend 的 `headwise_topk_softmax=True` 分支走纯 PyTorch：`scores_pqh = torch.einsum("thd,shd->ths", ...)` materialize `[T, h_q, S]` fp32 → 长序列 OOM。应改走 fused kernel（正确性优先，不抠速度）。

参考文档 `hsa_compressedlmkk_37eabce0b_semantics.md` 第9点**不在 repo**（群里发的，本地只有同名 config JSON）。需直接开 chunk prefill 跑 compare 复现。
